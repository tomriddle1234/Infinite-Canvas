# Media Preview Performance Plan - 2026-06-12

## Context

This plan covers only the canvas media preview optimization discussed on 2026-06-12.

Template sync is out of scope for this plan. The upstream prompt template markdown does not add new template categories compared with our current `static/system-prompts/templates/composition.json`; the missing template file noted during review is expected to exist on another machine as our own asset.

Third-party platform integrations, Jimeng, ComfyUI, Agnes, RunningHub, and the upstream standalone asset manager are also out of scope unless a specific change is required to support preview rendering.

## Current Implementation

Our current canvas output path renders original media URLs directly in the output grid:

- Images are rendered as normal `img src` elements.
- Videos are rendered as `video src` elements with `preload="metadata"`.
- Output nodes use grid layout and append/update helpers, which reduces unnecessary DOM reconstruction but does not reduce browser image decode, video metadata load, memory pressure, or GPU texture cost.

This means large canvases can still become slower when many generated images, videos, history thumbnails, or open output nodes are visible at once. The main cost is not only DOM count; it is decoding, resizing, painting, and keeping full-size media around.

## Is Preview Required?

Preview is not a hard functional requirement. The app can run without it.

Preview is a high-value performance optimization when users keep many images or videos on the canvas. It is most useful for:

- Large output nodes with many generated images.
- Video-heavy outputs.
- Saved canvases reopened with many historical outputs.
- Zoomed-out canvas browsing where full-resolution media is unnecessary.
- Lower-memory machines where many full-size images cause jank.

The recommended direction is to add preview as a progressive enhancement:

- Use lightweight preview URLs in grid thumbnails.
- Keep original URLs in `data-url` or metadata for lightbox/open/download actions.
- Fall back to the original URL if preview generation fails.

## Upstream Reference

The upstream implementation added a focused media preview API:

- `/api/media-preview?w=<size>&url=<encoded-media-url>`
- Image previews are generated through Pillow thumbnailing.
- Video previews are generated by extracting a frame with `ffmpeg`.
- Generated previews are cached by source path, modified time, file size, and requested width.
- Frontend helpers convert local `/output/` or `/assets/` media URLs into preview URLs for thumbnail display.

We should borrow the idea, not the surrounding asset-manager work.

## Proposed Scope

### Phase 1: Minimal Python Implementation

Add a preview endpoint to the current Python app path:

- Route: `GET /api/media-preview`
- Query params:
  - `url`: local media URL, limited to existing safe local media paths.
  - `w`: requested preview width, clamped to a safe range such as `64..2048`.
- Supported sources:
  - `/output/...`
  - `/assets/...`
  - Any other local media URL already accepted by the app's existing safe path resolver.
- Supported image formats:
  - `png`, `jpg`, `jpeg`, `webp`, and other formats already supported by Pillow if safe.
- Supported video formats:
  - `mp4`, `webm`, `mov`, `m4v`, only if `ffmpeg` is available.
- Cache directory:
  - Prefer a generated/cache folder under existing app data or output cache conventions.
  - Cache key should include absolute source path, file size, modified timestamp, and requested width.
- Response:
  - Prefer WebP for compact previews.
  - Fall back to PNG when WebP save fails.
  - Return 404 for missing source.
  - Return 415 for unsupported preview generation.

### Phase 2: Go Parity

Because the project is migrating to Go, add the same route to `app-go` before treating the feature as complete for the default runtime.

Go implementation options:

- For images, either use Go image decoders for common formats or call a small internal helper if parity is easier.
- For videos, call `ffmpeg` when available, with timeout and safe argument passing.
- Match Python route behavior closely so frontend code does not care which backend is running.

### Phase 3: Frontend Integration

Add small frontend helpers in `static/canvas.html` and mirrored Go static assets:

- `canvasMediaPreviewUrl(itemOrUrl, size = 512)`
- `canvasPreviewImageAttrs(originalUrl, size = 512)` if helpful

Use preview URLs in thumbnail contexts only:

- Output grid thumbnails.
- Log/history thumbnail strips if they currently render original media.
- Input source thumb rows where the source is local output or asset media.

Do not use previews for:

- Lightbox full preview.
- Download/open original.
- Generation references.
- Copy URL actions.

For videos:

- Display preview image in grid thumbnail where possible.
- Preserve original video URL for click/lightbox playback.
- If preview fails, fall back to current `<video preload="metadata">` behavior.

## Safety Rules

- Do not preview arbitrary external URLs.
- Do not accept `file://` paths or path traversal from the browser.
- Only resolve URLs through existing local media path helpers.
- Clamp preview dimensions.
- Put timeouts on video frame extraction.
- Avoid regenerating preview files if a valid cache file already exists.
- Do not delete original media.

## Validation Plan

Use a staged validation approach:

1. Import check:
   - Python: run the app import check under `OFX_dev`.
   - Go: run targeted Go tests/build checks if the Go route is implemented.
2. Endpoint checks:
   - Existing local image returns `200` with image content.
   - Existing local video returns `200` when `ffmpeg` is available.
   - Missing file returns `404`.
   - Unsupported file returns `415`.
3. Browser checks:
   - Output thumbnails render from `/api/media-preview`.
   - Clicking a thumbnail still opens the original full image/video.
   - Downloads and generation references still use original URLs.
   - A canvas with many images feels smoother when reopened or zoomed out.

## Recommendation

Implement this before any broad upstream asset-library sync. It is narrow, user-visible, and directly addresses canvas performance without pulling in unrelated upstream platform or asset-manager complexity.
